# Phase 4D: Storage Migrations

## Overview

The Melodarr application natively runs on Linux and Docker. As part of migrating from the legacy Lidarr branding, Melodarr updates the default application data paths and database names from `lidarr` to `melodarr` while ensuring backward compatibility with existing installations.

This document details how legacy configuration folders and databases are safely and idempotently migrated in Melodarr.

## Scope

- **Linux and Docker Native Targets**: The migration targets the `.config/Lidarr` pattern standard on Linux native and within Docker environments. 
- Windows AppData migrations and macOS Application Support fallback methods are **out of scope**. 

## Migration Behaviors

All migrations happen automatically during early initialization if `Melodarr` runs and detects existing `Lidarr` state without preexisting `Melodarr` state.

### 1. Application Data Folder Migration
The default user application data folder is automatically mapped based on the environment context.

- **Operation**: Bootstrapper logic intercepts the `AppDataFolder` configuration. If `~/.config/Melodarr` does not exist, but `~/.config/Lidarr` does exist, Melodarr executes an idempotent `Directory.Move(legacyLidarrPath, targetMelodarrPath)`.
- **Fallback Behavior**: In contexts where directories cannot be renamed (e.g. cross-volume links, restrictive Docker bind mounts), the migration attempts the rename, catches `Exceptions`, and simply sets the `AppDataFolder` property back to the legacy `Lidarr` path. 

### 2. Database Migration
Regardless of whether the application data folder is successfully renamed or falls back to using the old `Lidarr` path, the database files are renamed to native `melodarr` paths.

- **Operation**: Searches the resolved `AppDataFolder` for `lidarr.db`. If `melodarr.db` is not present but `lidarr.db` is, Melodarr will iteratively rename the core database and supporting journaling files (`-journal`, `-shm`, `-wal`) into `melodarr.db{ext}`.
- **Fail-safe**: The database renaming relies on `File.Move`, maintaining database integrity inside the app folder without manual user intervention.

### 3. Backup Compatibility
The naming schema for built-in application backups and restores natively prefers Melodarr but explicitly bridges the legacy name.

- **Creation**: Backups generated by `BackupService` generate as `melodarr_backup_v{Version}_{Date}.zip`. Temp tracking targets `melodarr_backup` folders.
- **Restoration**: The `BackupFileRegex` pattern was broadened: `^(lidarr|melodarr)_backup_(v[0-9.]+_)?[._0-9]+\.zip`. During the zip extraction phase, any `lidarr.db` file present in a zip payload is automatically extracted and restored as `melodarr.db` in the application data folder.

## Outcomes
These changes securely bridge Melodarr's new product identity and storage footprint with legacy operations, completing Phase 4D. No complex database rewriting or disruptive namespace restructuring was required to establish seamless interoperability.
